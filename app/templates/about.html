About this Project

The raw data set used in this Data Science project was the UK MOT data set. This is part of the UK government's open data, and is accessible here. The purpose of the MOT test is to ensure that all vehicles over a prescribed age are checked at least once a year to see that they comply with key roadworthiness and environmental requirements as prescribed in the law. After each test, an entry is made detailing the car details, test outcome and the fault(s) detected. The vehicle testing data currently available on the government website spans 9 years, from 2005 to 2013. In this raw form, it doesn't tell us much.

This project was part of my MSc Data Science programme at Lancaster University. This website is actually just an interactive way of displaying the results of an extensive, targetted analysis of the UK MOT vehicle testing data set. It is aimed at providing information on the pass rates, top faults discovered at MOT test failures, and the actual detailed descriptions of these faults.

Data Integration

The first main step in this project was data integration. As the documentation for the data wasn't so clear, much of the ideas for the data integration came only after much playing around with the data, inspection and manipulation of the data using command line tools, and in some cases, loading small chunks up into databases and running SQL scripts on them (call it trial and error, if you like).

Once the structure of the data was clear and the general direction of the analysis was established, work began on the aggregating of the data for the 9 years (2005 to 2013 inclusive), and the resulting data was over 800 million rows in total. Due to storage constraints on local machines, a high-end AWS PostgreSQL server had to be provisioned to handle this.

Data Analysis

This was the part of the project that actually answered the research questions. For example, we wanted to know the pass rates for the different models, so given that we had the entire test result data at hand, we could group by make, model, year, test outcome (pass or fail). But calculating the top faults wasn't as trivial. The information about the car make and models tested was in one table, while the information about the test items was in another, and the information about the parent faults and actual descriptions were in different other tables.

So we needed to get all the bits and pieces from each table into one table with all the fields of interest. Once this was successfully accomplished, we could carry out similar grouping, by make, model, year, different hierarchies of the faults, and actual fault descriptions.

Data Presentation

So finally we had actual tables that could give us the information we needed! But how would we make this accessible to the public, especially to non-technical people who needed the information? That is the purpose this website serves. It uses the final output tables from the analysis stage as a data source, but in a slightly different way from how tradional websites do this.
